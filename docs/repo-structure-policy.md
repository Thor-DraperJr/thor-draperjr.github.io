# Repo Structure And Pruning Policy

This repo should make the difference between publishable source, reusable code, durable evidence, and local scratch output obvious. When those categories blur, future agent sessions start from noisy `git status` output and stale assumptions.

## Source Of Truth

| Area | Path | Policy |
|---|---|---|
| Astro app | `astro-site/` | The site lives here. Build, preview, and audit commands run from this directory unless a workflow says otherwise. |
| Posts | `astro-site/src/content/posts/` | Markdown source for published and draft posts. New posts use `YYYY-MM-DD-slug.md`. |
| Reusable site components | `astro-site/src/components/` | Shared page, resume, deck, and visualization components. Keep broadly reusable components here. |
| Article-specific visuals | `astro-site/src/components/` for now | One-off article visuals may live here until there is enough volume to justify an article-scoped component folder. Name them by article or concept, not generic UI terms. |
| Runtime data/helpers | `astro-site/src/lib/` | Typed data and helper modules used by pages/components. |
| Public assets | `astro-site/public/assets/images/posts/<post-slug>/` | Images that are served by the site. Use one folder per post or visual project. |
| Native asset extracts | `astro-site/public/assets/images/posts/<post-slug>/native-assets/` | Crops, logos, and assets used by native components. Do not leave placeholder logos or letter tiles as silent defaults. |
| Prompts/agents/instructions | `.github/` | Durable workflow customizations that should travel with the repo. Keep broad rules in `copilot-instructions.md`; move specialized workflows into prompts or agents. |
| Durable review reports | `.github/reviews/` | Human-readable review summaries worth preserving across sessions. |

## Generated Artifacts

Generated local output should go under ignored paths by default.

| Artifact | Default path | Commit policy |
|---|---|---|
| Local scratch/review output | `astro-site/.artifacts/` | Ignored. Promote only the final human-readable summary or selected screenshots. |
| Walking deck audit screenshots/reports | `astro-site/deck-audit/` | Ignored. Open screenshots locally for proof, but do not commit routine runs. |
| All Aboard audit screenshots/reports | `astro-site/.artifacts/all-aboard-audit/` | Ignored. Use for iterative proof. Promote selected evidence only when there is a clear reason. |
| Legacy All Aboard audit evidence | `astro-site/all-aboard-audit/` | Tracked historical evidence exists here. Do not add new routine runs to this folder. Prune or move it only in a dedicated cleanup pass. |
| Build output | `astro-site/dist/` | Ignored. Generated by `npm run build`. |
| Astro generated types/cache | `astro-site/.astro/` | Ignored. |

If a generated artifact becomes important evidence, prefer a short written report in `.github/reviews/` over committing a large folder of screenshots. Commit screenshots only when the image itself is the evidence future readers need.

## Publishing And Pages

The repo contains `.github/workflows/deploy-astro.yml`, which builds `astro-site/` and deploys the `astro-site/dist` artifact through GitHub Actions.

GitHub Pages repository settings must match that model: Pages source should be GitHub Actions, not a legacy branch build from `main` `/`. If GitHub shows a separate `pages build and deployment` run that tries to parse the repo root with Jekyll, that is a repository settings issue, not an Astro code issue. Do not paper it over with a root `.nojekyll` unless the Pages source has been confirmed.

## Copilot Instruction Scope

`.github/copilot-instructions.md` should hold broad rules that apply to most work in this repo: no code bloat, rendered visual inspection, placeholder discipline, owner-confirmed facts, and the issue-to-plan workflow.

Reusable workflows should live elsewhere:

- Issue-start planning belongs in `.github/prompts/issue-planning.prompt.md`.
- General article review and creation belongs in `.github/prompts/article-pass.prompt.md`.
- Informative article graphics and animated infographic workflows belong in `.github/prompts/visual-storytelling.prompt.md`.
- Role-specific review behavior belongs in `.github/agents/`.
- Generated review outputs belong in `.github/reviews/` only when they are durable written summaries.
- Temporary harness notes, one-off audit commands, and article-specific cleanup notes should not accumulate as permanent global instructions.

Current review result: the broad visual QA and issue-to-plan rules should stay global. The durable visual workflow is about article graphics and visual storytelling across posts; single-post workflows should not become durable prompt commands unless the same process has proven useful across multiple posts or decks.

## Cleanup Checklist

Run this before publish commits for major visual/content work:

1. Run `npm run build` from `astro-site/`.
2. Render visual pages/components in the browser and inspect screenshots when the change affects layout, motion, or embedded graphics.
3. Confirm generated output stayed in ignored paths such as `astro-site/.artifacts/` or `astro-site/deck-audit/`.
4. Check `git status --short` from the repo root.
5. Keep source, reusable code, public assets, and durable written reviews.
6. Remove or leave ignored local scratch artifacts uncommitted.
7. For article-specific assets, verify every referenced file is used by the post/component or intentionally retained as source/reference material.
8. For placeholder logos, monograms, or letter tiles, either replace them with real assets or document them as active TODOs before publishing.
9. After publish workflow changes, verify Pages still uses GitHub Actions as the source through the GitHub UI or an authenticated GitHub API/CLI call.

## When To Split Structure Further

Do not add folders just to feel organized. Split only when the current structure creates real friction:

- Move article-specific components out of `src/components/` only after several one-off visuals make the directory hard to scan.
- Add a dedicated evidence folder only when `.github/reviews/` cannot represent the proof clearly.
- Promote scripts into a repeatable tool or skill only after the same workflow has been run successfully on more than one article/deck.
